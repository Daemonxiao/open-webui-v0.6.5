# Prompt 应用权限与使用 PRD

## 1. 背景

当前系统已有 Workspace Prompts 能力，Prompt 主要由 `name`、`command`、`content`、`tags`、`access_grants`、`is_active`、`version_id` 等字段组成。现有实现更接近“可共享的 Prompt 模板”，读权限用户可以在列表、详情、复制、历史等入口看到完整 `content`。

本需求不再兼容旧 Workspace Prompt 功能。旧 Workspace Prompt 的列表、详情、复制、导出、`/` 命令插入等能力需要下线或替换为 Prompt 应用能力，避免继续暴露指令。

本需求希望把 Prompt 升级为面向终端用户可直接使用的“Prompt 应用”。本期先收敛为管理员创建和管理，权限模型需要预留后续扩展：

- 管理员可以创建、编辑、查看、删除自己的 Prompt 应用。
- 管理员可以查看并管理所有人的 Prompt 应用。
- 其他人可以使用管理员创建的 Prompt 应用。
- 其他人不能对该 Prompt 应用做新增、查看指令、编辑、删除等管理动作。
- 其他人不能看到 Prompt 的具体指令细节。
- 后续可以扩展为每个人都能创建、编辑、查看、删除自己的 Prompt 应用。
- 后续每个人只能管理自己的 Prompt 应用，管理员仍可以查看并管理所有人的 Prompt 应用。

## 2. 目标

### 2.1 产品目标

提供一个由创建者维护、其他用户可调用的 Prompt 应用体系。本期创建者限定为管理员；管理员把常用业务场景封装为 Prompt 应用，其他用户只看到名称和描述，并通过聊天输入或应用入口使用，不接触底层指令。

权限模型需要支持后续平滑升级：创建能力从“仅管理员”放开为“所有人”，普通用户的管理能力始终绑定 owner，管理员保留全量管理能力。

### 2.2 非目标

- 不做多用户协作编辑。普通用户之间不能共同编辑同一个 Prompt 应用；管理员的全量管理属于平台治理权限。
- 本期不开放普通用户创建 Prompt 应用，但数据模型和权限判断要预留后续开放。
- 不向普通用户展示 Prompt 指令、历史版本、导出内容或复制指令。
- 不在本期做 Prompt 应用市场、评分、收藏、分类运营等能力。
- 不保留旧 Workspace Prompt 的用户侧使用方式，包括通过 `/` 命令把 Prompt 指令插入输入框。

## 3. 术语

- Prompt 应用：用户创建的一条可复用 Prompt 配置。本期创建者限定为管理员。
- 指令：Prompt 应用实际注入模型的系统/用户指令内容，对普通用户隐藏。
- 管理动作：创建、详情查看、编辑、删除、启用/停用、版本管理、访问范围配置。
- 使用动作：选择 Prompt 应用，并把用户输入与隐藏指令组合后发起聊天。

## 4. 角色与权限

| 角色 | 创建 | 管理自己创建的应用 | 管理他人创建的应用 | 使用可用应用 | 查看指令细节 |
| --- | --- | --- | --- | --- | --- |
| 管理员 owner | 是 | 是 | 是 | 是 | 是 |
| 管理员非 owner | 是 | 是 | 是 | 是 | 是 |
| 普通用户 | 否 | 否 | 否 | 是 | 否 |

说明：

- 管理员拥有 Prompt 应用的全量管理权限，包括查看指令、编辑、删除、启用/停用、版本管理和访问范围配置。
- 后续放开普通用户创建后，普通用户只能管理自己创建的应用。
- 后续管理员仍可以管理所有用户创建的 Prompt 应用。
- “使用”不等于读权限。非管理者可以看到应用名称、描述、创建者、启用状态，但不能获取指令原文。
- 如果未来需要团队协作编辑，应另开需求设计 owner 转移、共同维护者和审计。

## 5. Prompt 应用字段

### 5.1 必填字段

- 名称：面向用户展示，例如“合同条款风险检查”。
- 描述：告诉用户这个应用适合什么场景、需要输入什么信息。
- 指令：实际 Prompt 内容，仅有管理权限的人可见。本期包括所有管理员；后续包括 owner 本人和所有管理员。

### 5.2 系统字段

- ID：系统生成。
- 创建者 user_id：用于 owner 判断。
- 创建者角色：可选记录创建时用户角色，用于审计和筛选。
- 启用状态：启用后可被用户使用，停用后所有非管理者不可见或不可用。
- 创建时间、更新时间。
- 版本信息：如果沿用现有 Prompt History，版本内容只对有管理权限的人可见。

### 5.3 字段映射建议

当前代码已有 `name` 和 `content`，但缺少独立 `description`。

- `name` 映射“名称”。
- 新增 `description` 字段。
- `content` 映射“指令”。

建议优先新增显式 `description` 字段，避免后续列表查询、搜索、校验、导入导出都依赖松散 `meta`。

## 6. 使用流程

### 6.1 管理员创建 Prompt 应用

1. 管理员进入 Prompt 应用管理页。
2. 点击新建。
3. 填写名称、描述、指令。
4. 保存后应用默认启用。
5. 系统只把该应用的管理权限授予创建者。

校验：

- 名称不能为空。
- 描述不能为空，建议 10 到 300 字。
- 指令不能为空。
- 同一 owner 下名称不建议重复；如保留 command，则 command 全局唯一或自动生成唯一 slug。

后续扩展：

- 普通用户也可以进入自己的 Prompt 应用管理页。
- 普通用户创建后成为 owner，只能管理自己创建的应用。
- 管理员可以在管理端看到并管理所有人创建的应用。

### 6.2 编辑 Prompt 应用

1. 本期管理员在管理列表中可以看到所有 Prompt 应用。
2. 进入详情后可以查看和修改名称、描述、指令。
3. 保存后记录更新时间；如沿用历史版本，记录版本快照。
4. 管理者可以启用/停用自己有管理权限的应用。

限制：

- 普通用户不能进入或修改他人创建的 Prompt 应用详情。
- 普通用户不能删除他人创建的 Prompt 应用。
- 普通用户不能通过 API 修改、删除、切换他人应用。
- 后续放开普通用户创建后，普通用户不能管理他人应用；管理员可以管理所有应用。

### 6.3 删除 Prompt 应用

1. 管理者在自己有管理权限的应用列表点击删除。
2. 系统二次确认。
3. 删除后普通用户不能再搜索或使用。

建议：

- 本期可沿用硬删除。
- 如果未来需要审计或恢复，再改为软删除。

### 6.4 非管理者使用 Prompt 应用

1. 非管理者在聊天输入区、Prompt 应用选择器或应用列表中看到可用应用。
2. 非管理者只能看到名称和描述。
3. 用户选择应用后输入业务内容。
4. 前端提交应用 ID 和用户输入。
5. 后端读取隐藏指令，把指令与用户输入组合后调用模型。
6. 聊天记录中可以展示使用了哪个 Prompt 应用，但不展示指令内容。

用户可见信息：

- 名称。
- 描述。
- 创建者名称或管理员标识。
- 是否可用。

用户不可见信息：

- 指令原文。
- 历史版本指令。
- 导出文件中的指令。
- 复制指令按钮。
- 编辑入口。
- 删除入口。
- 访问控制入口。

## 7. 页面与交互要求

### 7.1 管理端列表

入口：Prompt 应用管理页。

列表展示：

- 名称。
- 描述。
- 启用状态。
- 更新时间。
- 操作：编辑、启用/停用、删除。

过滤规则：

- 本期管理员管理端显示所有 Prompt 应用。
- 后续开放个人创建后，普通用户只看到自己创建的 Prompt 应用。
- 后续管理员管理端可以增加“全部”“我创建的”“普通用户创建的”等筛选视图。

### 7.2 管理端详情/编辑页

字段：

- 名称输入框。
- 描述文本框。
- 指令文本框。
- 保存按钮。
- 删除按钮。
- 启用/停用开关。

如果沿用现有版本历史：

- 版本列表只对有管理权限的人可见。
- 历史版本中的指令内容只对有管理权限的人可见。

### 7.3 用户端使用入口

入口建议：

- 聊天输入框的 Prompt 选择菜单。
- 或一个轻量的“Prompt 应用”选择面板。

展示：

- 名称。
- 描述。

交互：

- 点击应用后，应用被选中。
- 输入框可以提示“正在使用：{名称}”。
- 用户发送消息时，系统自动附加隐藏指令。
- 用户可以取消选择应用。

禁止：

- 不提供复制指令。
- 不跳转到管理详情。
- 不展示 Prompt 内容 tooltip。

## 8. API 要求

### 8.1 管理 API

管理 API 必须使用统一的 `can_manage_prompt` 校验，避免把 admin 全局权限写死在各接口中：

- `POST /prompts/create`：本期仅管理员可创建；后续可配置为所有人可创建。
- `GET /prompts/admin/list`：返回所有 Prompt 应用，仅管理员可用。
- `GET /prompts/my/list`：后续可用于返回当前用户自己创建的 Prompt 应用。
- `GET /prompts/id/{id}`：仅有管理权限的人可返回完整指令。
- `POST /prompts/id/{id}/update`：仅有管理权限的人可更新。
- `POST /prompts/id/{id}/toggle`：仅有管理权限的人可切换启用状态。
- `DELETE /prompts/id/{id}/delete`：仅有管理权限的人可删除。
- History、diff、export、access update 等接口同样仅有管理权限的人可用。

### 8.2 使用 API

需要区分“可使用摘要”和“完整管理详情”。

建议新增：

- `GET /prompts/apps`：返回当前用户可使用的 Prompt 应用摘要。
- `POST /prompts/apps/{id}/run`：普通用户提交输入并使用该 Prompt 应用。

`GET /prompts/apps` 返回字段：

```json
{
  "id": "prompt-id",
  "name": "合同条款风险检查",
  "description": "检查合同条款中的付款、违约、终止和责任风险。",
  "owner": {
    "name": "Admin"
  },
  "is_active": true
}
```

不得返回：

- `content`
- `access_grants`
- `version_id`
- history snapshot
- prompt diff

`POST /prompts/apps/{id}/run` 请求示例：

```json
{
  "input": "请检查这段合同条款：..."
}
```

后端行为：

- 校验应用存在且启用。
- 校验用户有使用权限。
- 后端读取指令。
- 后端把指令和用户输入组装进聊天请求。
- 响应模型输出或创建对应聊天消息。

## 9. 权限实现原则

### 9.1 管理权限

本期管理权限：

```text
can_manage = user.role == "admin"
```

后续扩展后的管理权限：

```text
can_manage =
  user.role == "admin"
  or prompt.user_id == user.id
```

关键约束：

- 管理员拥有全量管理权限。
- 普通用户管理自己创建的应用通过 `prompt.user_id == user.id` 判断。
- 普通用户永远不能管理他人创建的应用。
- 所有管理接口都必须调用统一的 `can_manage_prompt`，不要在不同接口里手写不同条件。

### 9.2 使用权限

使用权限可以先简单定义为：

```text
can_use = prompt.is_active == true and user is verified
```

如果后续需要分组开放，可再引入 `access_grants` 的 `use` 权限。但不要复用现有 `read` 权限表达“可使用”，因为 read 容易被理解为可读指令内容。

### 9.3 指令保护

指令保护必须在后端完成，不能只靠前端隐藏。

需要删除或替换的旧泄露入口：

- 列表接口返回 `content`。
- 详情接口返回 `content`。
- 命令选择接口返回完整 Prompt。
- 复制 Prompt 按钮。
- 导出 Prompt。
- Prompt history。
- Prompt diff。
- 浏览器网络响应。
- 聊天记录中的消息 metadata。

## 10. 与现有实现的差距

当前代码中需要特别注意：

- `PromptModel.content` 是必返字段，普通 read 用户会拿到完整指令。
- `GET /api/v1/prompts/` 被聊天输入区 Prompt 命令菜单使用，会返回完整 Prompt。
- Prompt 列表页展示 `prompt.content`，并提供复制。
- `get_prompt_by_id`、`get_prompt_by_command` 对 admin 直接放行。
- update/delete/toggle/history 等接口允许非 owner admin 操作，这一点符合管理员全量管理要求，但需要统一收敛到 `can_manage_prompt`。
- `BYPASS_ADMIN_ACCESS_CONTROL` 会让 admin 看到全部 Prompt，这一点符合管理员全量管理要求，但普通用户路径仍必须严格脱敏。
- 现有 `access_grants read` 不适合表达“可使用但不可看指令”。
- 后续如果开放普通用户创建，需要确认用户侧管理列表和管理员全量管理列表分离。
- 旧 Workspace Prompt 用户侧入口不需要兼容，应直接移除或改为 Prompt 应用选择入口。

## 11. 验收标准

### 11.1 管理员 owner

- 可以创建 Prompt 应用。
- 可以在管理列表看到所有人的 Prompt 应用。
- 可以查看名称、描述、指令。
- 可以编辑名称、描述、指令。
- 可以启用/停用。
- 可以删除。

### 11.2 管理员非 owner

- 可以在管理列表看到所有人的 Prompt 应用。
- 可以通过详情接口获取任意 Prompt 应用的完整指令。
- 可以编辑、删除、启用/停用任意 Prompt 应用。
- 可以管理任意 Prompt 应用的版本、历史和访问范围。

### 11.3 普通用户

- 可以看到已启用 Prompt 应用的名称和描述。
- 可以选择应用并发起聊天。
- 不能看到指令原文。
- 不能复制、导出、编辑、删除应用。
- 不能访问历史版本、diff 或管理详情。

### 11.4 后续扩展验收

- 普通用户可以创建自己的 Prompt 应用。
- 普通用户可以编辑、查看、删除、启用/停用自己创建的 Prompt 应用。
- 普通用户不能管理其他人创建的 Prompt 应用。
- 管理员可以管理所有用户创建的 Prompt 应用。
- 非管理者使用应用时仍不能看到指令原文。

### 11.5 停用状态

- 有管理权限的人仍可在管理端看到停用应用。
- 非管理者不能在使用入口看到停用应用。
- 非管理者不能通过 run API 使用停用应用。

## 12. 测试建议

- 后端单测覆盖 owner admin、非 owner admin、普通用户三类身份。
- 后端单测验证所有管理接口都走统一 `can_manage_prompt`。
- 后续扩展时补充普通用户 owner、普通用户不能管理他人应用、管理员可以管理所有应用三类测试。
- 后端单测验证使用列表不返回 `content`。
- 后端单测验证 run API 可以使用隐藏指令，但响应不泄露指令。
- 前端测试或手工验证普通用户网络响应中没有 `content`、history snapshot、diff。
- 回归测试现有 Prompt 创建、编辑、删除、聊天选择入口。

## 13. 发布策略

建议分两步发布：

1. 后端先落权限和脱敏 API，保证即使旧前端入口存在也不会泄露指令。
2. 前端再调整管理页和用户使用入口，移除普通用户复制、导出、详情、历史等入口。

数据处理：

- 当前没有需要保留的历史 Prompt 数据，不做历史数据迁移。
- 只需要保证新建 Prompt 应用时写入名称、描述、指令和创建者。
